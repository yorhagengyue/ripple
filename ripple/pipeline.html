// 02 / Pipeline · powered by Workato

Three beats. One spine.

Workato is not middleware here — it is the agent's nervous system. Every sample enters through a Workato webhook. Every decision is a Workato recipe. And when any AI client — Claude Desktop, Claude Code, Codex, Cursor — needs live physiological context, Workato flips role: it exposes itself as a Model Context Protocol server so the LLM dials into the same recipes as callable tools. One source of truth, zero shadow state.

  1. 01
    Sense · Ingest

    Watch → Supabase, every five minutes.

    The Apple Watch writes heart rate, HRV, sleep, and motion into HealthKit. Health Auto Export packages them into a JSON batch and pushes it to a Workato webhook. Recipe 1 normalises the metric name and upserts every sample into healthlog with the original timestamp — no aggregation, no editorialising.

    TriggerHTTP webhook · ~5 min cadence Writeshealthlog (Supabase / PostgREST) Recipe01 · ripple_ingest_webhook
  2. 02
    Think · Daily watchdog

    Once a day, a slow look across everything.

    Every twenty-four hours Recipe 6 recomputes each metric's seven-day mean against its thirty-day baseline and refreshes the baseline view. When enough metrics drift in the same direction it writes an intervention row with a risk level (1–3) and an AI-generated summary — the same row any MCP client can read back via get_recent_anomaly_log.

    Triggercron · 24 h Readshealthlog (30 d) Writesbaseline · intervention Recipe06 · ripple_anomaly_watchdog
  3. 03
    MCP
    Tools · Workato as an MCP server

    Workato exposes itself. Kimi dials in.

    This is the part that makes Ripple an agent and not just a webhook fan-out. Four Workato recipes are published as Model Context Protocol tools — including get_current_vitals and get_baseline_deviation. When the user asks "how's my heart rate" mid-chat, Kimi does not hallucinate from its system prompt — it issues an MCP tool call, Workato runs the corresponding recipe (reads Supabase, shapes the payload), and Kimi returns a grounded answer inside the same turn.

    The Chat tab and the MCP agent hit the same recipes. No shadow SQL, no duplicated prompts, no drift between "what the human chat shows" and "what the agent acts on." Workato is simultaneously the orchestrator and the tool surface.

    Connect

    Wire it into any MCP client.

    Because Workato speaks MCP, the same four tools light up in every MCP-compatible agent — Claude Desktop, Claude Code, Codex CLI, Cursor, Continue, MCP Inspector. Pick your client, paste the config, restart the client, and ripple-wellness appears in your tool list.

    Endpoint https://ripple-wellness.vercel.app/api/mcp

    Transport: mcp-remote → HTTPS, native endpoint /api/mcp — public read-only demo spine, no token.

    Three shell-ready commands that install ripple-wellness on any machine. Pick the one that matches your client, paste, done. No manual JSON editing.
    Claude Code — any terminal
    claude mcp add ripple-wellness --transport stdio -- npx -y mcp-remote "https://ripple-wellness.vercel.app/api/mcp"
    Codex CLI — appends to ~/.codex/config.toml
    cat >> ~/.codex/config.toml <<'EOF'

[mcp_servers.ripple-wellness]
command = "npx"
args = ["-y", "mcp-remote", "https://ripple-wellness.vercel.app/api/mcp"]
EOF
    Claude Desktop — Python merges the JSON safely
    python3 <<'EOF'
import json, pathlib
p = pathlib.Path.home() / "Library/Application Support/Claude/claude_desktop_config.json"
cfg = json.loads(p.read_text()) if p.exists() else {}
cfg.setdefault("mcpServers", {})["ripple-wellness"] = {
    "command": "npx",
    "args": ["-y", "mcp-remote",
             "https://ripple-wellness.vercel.app/api/mcp"]
}
p.write_text(json.dumps(cfg, indent=2))
print("Installed. Restart Claude Desktop.")
EOF
    After install, run /mcp (Claude Code) or click the 🔌 menu (Desktop) — ripple-wellness shows up with get_current_vitals + get_baseline_deviation ready to call.
